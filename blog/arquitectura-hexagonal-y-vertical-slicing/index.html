Arquitectura hexagonal + vertical slicing en un API real

Cómo estructuro un API para que crezca sin volverse opaca: un caso de uso de punta a punta, sin teoría de más.

21 jun 2026 · 9 min de lectura · Wilvardo Ramírez

La arquitectura por capas tradicional te ordena el código hasta que el proyecto crece. Entonces te das cuenta de que ordené las carpetas, no las decisiones.

Una estructura puede verse impecable —controllers, services, models, cada cosa en su cajón— y aun así no decirte nada sobre qué hace el sistema. Ordenas por tecnología y pierdes la intención. Este tutorial trata de cómo estructuro un API para que crezca sin volverse opaca: un caso de uso de punta a punta, sin teoría de más. El caso será registrar un cliente. Limpio, de negocio, sin autenticación ni nada que distraiga del patrón. El patrón es el protagonista.

El problema de las capas cuando el dominio crece

La arquitectura en capas agrupa por tipo técnico. Todos los controladores juntos. Todos los servicios juntos. Todos los modelos juntos. El primer mes es cómodo: sabes dónde va cada archivo nuevo.

El problema aparece cuando el dominio crece. Para entender una sola funcionalidad —registrar un cliente— tienes que abrir el controlador en una carpeta, el servicio en otra, el modelo en una tercera y el repositorio en una cuarta. La lógica de una intención está repartida en cuatro lugares por una razón puramente técnica: cada archivo es de un "tipo". Nada de eso te dice qué hace el sistema.

Y el acoplamiento corre por la capa equivocada. Tu servicio importa directo la implementación de la base de datos. El dominio sabe que hay MySQL abajo. Cambiar de motor, o probar la lógica sin tocar la base, deja de ser una decisión y se vuelve una cirugía. Las capas técnicas ordenan los archivos por lo que son, no por lo que hacen. Y un sistema se mantiene por lo que hace.

Hexagonal + vertical slicing en una frase

Lo resumo así: organiza el código por caso de uso y aísla el dominio de la infraestructura.

Vertical slicing es la primera mitad. En lugar de cortar el sistema en capas horizontales por tecnología, lo cortas en rebanadas verticales por intención. Cada rebanada —registrar un cliente, emitir una factura, cancelar un pedido— contiene todo lo suyo junto. Abres una carpeta y ves la funcionalidad completa, no un fragmento.

Hexagonal es la segunda. El dominio —las reglas del negocio— vive en el centro y no conoce el mundo exterior. No sabe si los datos llegan por HTTP ni si se guardan en MySQL o en memoria. Habla con el exterior a través de puertos: interfaces que el dominio define y que la infraestructura implementa. La base de datos y el framework son detalles enchufables, no cimientos. Por eso el dominio depende de una interfaz, nunca de una implementación. Esa inversión es la que mantiene el centro limpio.

La estructura de carpetas (una carpeta por caso de uso)

Así se ve en disco. Una carpeta por caso de uso, con sus capas adentro:

src/
  customers/
    register-customer/
      domain/
        customer.entity.ts                   # entidad de dominio
        customer-repository.port.ts          # puerto (interfaz)
        email-already-registered.error.ts    # error del dominio
      application/
        register-customer.dto.ts             # datos de entrada
        register-customer.use-case.ts        # orquesta el caso de uso
      infrastructure/
        in-memory-customer.repository.ts     # adaptador del puerto
        customer.controller.ts               # traduce HTTP <-> caso de uso
        register-customer.routes.ts          # rutas de este caso de uso
  shared/
    errors/
      app.error.ts                           # clase base de errores
    http/
      http-handler.ts                        # contrato de handler HTTP
  router.ts                                  # importa las rutas de cada caso

Mira el orden de dependencias: domain no importa nada de afuera. application importa de domain. infrastructure importa de ambos. Las flechas apuntan siempre hacia el centro, nunca al revés. Esa regla —que las dependencias solo apunten hacia adentro— es lo que hace que el patrón aguante cuando agregas la rebanada veinte.

Y un detalle que no es cosmético: un archivo de rutas por caso de uso, todos importados en el router principal. El router no concentra lógica. Solo ensambla. Cuando agregas una funcionalidad, agregas una carpeta y enchufas su archivo de rutas. No tocas el resto.

La rebanada completa, con código

Vamos de adentro hacia afuera. Primero el dominio, luego el caso de uso, después el puerto y su adaptador, y al final el controlador y las rutas. Mismos nombres, mismos archivos.

El stack es Node con TypeScript llano, sin framework. Es a propósito: si el patrón depende de Express o de Adonis, no es el patrón, es el framework. Esto se traslada a cualquiera de los dos cambiando solo la capa HTTP.

Arrancamos por la entidad. Es el corazón de la rebanada: representa un cliente y protege sus propias reglas. No sabe que existe una base de datos.

// src/customers/register-customer/domain/customer.entity.ts

/** Datos mínimos para construir un cliente válido. */
export interface CustomerProps {
  readonly id: string;
  readonly name: string;
  readonly email: string;
}

/**
 * Entidad de dominio. Garantiza que un cliente nunca exista
 * en un estado inválido: si se construye, es válido.
 */
export class Customer {
  private constructor(private readonly props: CustomerProps) {}

  /** Crea un cliente validando sus invariantes de negocio. */
  static create(props: CustomerProps): Customer {
    if (props.name.trim().length === 0) {
      throw new Error("El nombre del cliente es obligatorio.");
    }
    if (!Customer.isEmail(props.email)) {
      throw new Error("El correo del cliente no es válido.");
    }
    return new Customer({ ...props, email: props.email.toLowerCase() });
  }

  private static isEmail(value: string): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value);
  }

  get id(): string {
    return this.props.id;
  }

  get email(): string {
    return this.props.email;
  }

  /** Representación serializable para la capa de salida. */
  toPrimitives(): CustomerProps {
    return { ...this.props };
  }
}

Sigue el puerto: la interfaz que el caso de uso necesita para persistir. El dominio la define; el dominio decide qué necesita del mundo. Quién la implemente es problema de la infraestructura.

// src/customers/register-customer/domain/customer-repository.port.ts
import { Customer } from "./customer.entity";

/**
 * Puerto de persistencia. El dominio depende de esta interfaz,
 * no de ninguna base de datos concreta.
 */
export interface CustomerRepository {
  /** Devuelve el cliente con ese correo, o null si no existe. */
  findByEmail(email: string): Promise<Customer | null>;
  /** Persiste un cliente nuevo. */
  save(customer: Customer): Promise<void>;
}

Y el error de dominio. Que un correo ya esté registrado es una regla del negocio, no un fallo técnico. Por eso vive aquí, no en el controlador.

// src/customers/register-customer/domain/email-already-registered.error.ts
import { AppError } from "../../../shared/errors/app.error";

/** El correo ya pertenece a otro cliente. Conflicto de negocio. */
export class EmailAlreadyRegisteredError extends AppError {
  constructor(email: string) {
    super({
      title: "Correo ya registrado",
      detail: `Ya existe un cliente con el correo ${email}.`,
      status: 409,
    });
  }
}

Ahora el caso de uso. Orquesta: recibe datos, consulta el puerto, aplica la regla, persiste. No sabe de HTTP ni de SQL. Depende de la interfaz CustomerRepository, que le inyectan por el constructor. Esa inyección es la inversión de dependencias en acto.

// src/customers/register-customer/application/register-customer.dto.ts

/** Datos de entrada del caso de uso. Planos, sin lógica. */
export interface RegisterCustomerDto {
  readonly name: string;
  readonly email: string;
}

// src/customers/register-customer/application/register-customer.use-case.ts
import { randomUUID } from "node:crypto";
import { Customer } from "../domain/customer.entity";
import { CustomerRepository } from "../domain/customer-repository.port";
import { EmailAlreadyRegisteredError } from "../domain/email-already-registered.error";
import { RegisterCustomerDto } from "./register-customer.dto";

/** Caso de uso: registrar un cliente nuevo. */
export class RegisterCustomerUseCase {
  // Depende del puerto, no de la implementación.
  constructor(private readonly customers: CustomerRepository) {}

  /** Devuelve el id del cliente creado. */
  async execute(dto: RegisterCustomerDto): Promise<{ id: string }> {
    const existing = await this.customers.findByEmail(dto.email);
    if (existing !== null) {
      throw new EmailAlreadyRegisteredError(dto.email);
    }

    const customer = Customer.create({
      id: randomUUID(),
      name: dto.name,
      email: dto.email,
    });

    await this.customers.save(customer);
    return { id: customer.id };
  }
}

Falta el adaptador: una implementación concreta del puerto. Aquí es en memoria para que el ejemplo corra sin base de datos. El día que quieras MySQL, escribes mysql-customer.repository.ts con la misma interfaz y no cambia nada más. Eso es enchufar un detalle.

// src/customers/register-customer/infrastructure/in-memory-customer.repository.ts
import { Customer } from "../domain/customer.entity";
import { CustomerRepository } from "../domain/customer-repository.port";

/** Adaptador en memoria. Útil para pruebas y para arrancar. */
export class InMemoryCustomerRepository implements CustomerRepository {
  private readonly store = new Map<string, Customer>();

  async findByEmail(email: string): Promise<Customer | null> {
    const key = email.toLowerCase();
    return this.store.get(key) ?? null;
  }

  async save(customer: Customer): Promise<void> {
    this.store.set(customer.email, customer);
  }
}

Llegamos al controlador. Es delgado a propósito: traduce HTTP hacia el caso de uso y mapea el resultado —o el error— a una respuesta REST. Nada más. Para no atarlo a ningún framework, defino un contrato de handler explícito; es portable a Express, Adonis u otro framework, adaptando solo este borde.

// src/shared/http/http-handler.ts

/** Petición HTTP mínima, agnóstica del framework. */
export interface HttpRequest {
  readonly body: unknown; // sin tipar aún: se valida en el borde
}

/** Respuesta HTTP mínima, agnóstica del framework. */
export interface HttpResponse {
  status: number;
  body: unknown;
}

/** Contrato de un handler HTTP. */
export type HttpHandler = (req: HttpRequest) => Promise<HttpResponse>;

El body entra como unknown, no como any. La diferencia importa: unknown me obliga a validar antes de usarlo. Ese estrechamiento es el borde donde lo no confiable se vuelve confiable.

// src/customers/register-customer/infrastructure/customer.controller.ts
import { HttpHandler, HttpRequest, HttpResponse } from "../../../shared/http/http-handler";
import { AppError } from "../../../shared/errors/app.error";
import { RegisterCustomerUseCase } from "../application/register-customer.use-case";
import { RegisterCustomerDto } from "../application/register-customer.dto";

/** Construye el handler de "registrar cliente" con su caso de uso inyectado. */
export function makeRegisterCustomerController(
  useCase: RegisterCustomerUseCase,
): HttpHandler {
  return async (req: HttpRequest): Promise<HttpResponse> => {
    try {
      const dto = parseBody(req.body);
      const result = await useCase.execute(dto);
      return { status: 201, body: result };
    } catch (error: unknown) {
      return toHttpError(error);
    }
  };
}

/** Valida el cuerpo crudo y lo convierte en el DTO tipado. */
function parseBody(body: unknown): RegisterCustomerDto {
  if (typeof body !== "object" || body === null) {
    throw new AppError({ title: "Cuerpo inválido", detail: "Se esperaba un objeto.", status: 400 });
  }
  const { name, email } = body as Record<string, unknown>;
  if (typeof name !== "string" || typeof email !== "string") {
    throw new AppError({ title: "Campos inválidos", detail: "name y email deben ser texto.", status: 400 });
  }
  return { name, email };
}

/** Mapea cualquier error a la forma REST. Lo conocido, con su estado; lo demás, 500. */
function toHttpError(error: unknown): HttpResponse {
  if (error instanceof AppError) {
    return { status: error.status, body: error.toResponse() };
  }
  return {
    status: 500,
    body: { title: "Error interno", detail: "Algo salió mal.", key: "error-interno" },
  };
}

Y las rutas de este caso de uso, en su propio archivo. Aquí se arma la rebanada: se instancia el adaptador, se inyecta al caso de uso, se construye el handler. El cableado vive en el borde, no en el dominio.

// src/customers/register-customer/infrastructure/register-customer.routes.ts
import { InMemoryCustomerRepository } from "./in-memory-customer.repository";
import { RegisterCustomerUseCase } from "../application/register-customer.use-case";
import { makeRegisterCustomerController } from "./customer.controller";

const repository = new InMemoryCustomerRepository();
const useCase = new RegisterCustomerUseCase(repository);
const handler = makeRegisterCustomerController(useCase);

/** Una ruta = un método, una ruta y su handler. El router las ensambla. */
export const registerCustomerRoutes = [
  { method: "POST" as const, path: "/customers", handler },
];

El router principal solo junta las rebanadas. No conoce ningún caso de uso por dentro:

// src/router.ts
import { registerCustomerRoutes } from "./customers/register-customer/infrastructure/register-customer.routes";

/** Tabla de rutas del API. Cada caso de uso aporta las suyas. */
export const routes = [
  ...registerCustomerRoutes,
  // ...siguientes casos de uso
];

Los errores: título, detalle y key (y por qué el controlador es delgado)

Toda respuesta de error de mi API tiene la misma forma: title, detail y key. El título es legible. El detalle explica el caso puntual. La key es el slug del título: un identificador estable que el frontend puede usar para decidir qué hacer sin parsear texto que mañana cambia.

La clase base concentra esa forma una sola vez:

// src/shared/errors/app.error.ts

/** Forma estándar de un error de API: title, detail, key. */
export interface ErrorResponse {
  readonly title: string;
  readonly detail: string;
  readonly key: string;
}

interface AppErrorProps {
  readonly title: string;
  readonly detail: string;
  readonly status: number;
}

/** Error base de la aplicación. La key se deriva del título. */
export class AppError extends Error {
  readonly title: string;
  readonly detail: string;
  readonly status: number;

  constructor(props: AppErrorProps) {
    super(props.detail);
    this.title = props.title;
    this.detail = props.detail;
    this.status = props.status;
  }

  /** Convierte el título en slug: "Correo ya registrado" -> "correo-ya-registrado". */
  get key(): string {
    return this.title
      .toLowerCase()
      .normalize("NFD")
      .replace(/[\u0300-\u036f]/g, "") // quita acentos (marcas combinantes)
      .replace(/[^a-z0-9]+/g, "-")
      .replace(/(^-|-$)/g, "");
  }

  /** Cuerpo listo para la respuesta REST. */
  toResponse(): ErrorResponse {
    return { title: this.title, detail: this.detail, key: this.key };
  }
}

Fíjate dónde nace EmailAlreadyRegisteredError: en el dominio, no en el controlador. La regla "no puede haber dos clientes con el mismo correo" es del negocio. El controlador no la conoce; solo sabe que un AppError trae su propio estado y su propia forma, y lo traduce a HTTP.

Por eso el controlador es delgado. Cada vez que un controlador engorda —validaciones de negocio, decisiones, consultas— es señal de lógica que se filtró desde el dominio. Un controlador delgado no es elegancia. Es la prueba de que el dominio quedó del lado correcto. Si la regla vive en el centro, se puede probar sin levantar un servidor y reusar desde otra entrada —una cola, un cron— sin copiar nada.

Tu turno: replica el patrón

Toma un caso de uso tuyo. No "clientes": el siguiente que de verdad tengas que construir. Emitir una factura, cancelar una suscripción, mover un pedido de estado. Replica la rebanada completa: entidad y puerto en el dominio, caso de uso y DTO en aplicación, adaptador y controlador y rutas en infraestructura. Enchufa su archivo de rutas al router y no toques nada más.

Cuando termines, revísalo contra esta lista. Si algo falla, la rebanada está mal cortada:

  • ¿El dominio importa algo de infrastructure? Si sí, apuntaste una dependencia hacia el lado equivocado.
  • ¿El caso de uso menciona HTTP, SQL o el nombre de tu base de datos? Si sí, se filtró infraestructura al centro.
  • ¿Podrías cambiar de base de datos escribiendo solo un adaptador nuevo, sin tocar dominio ni caso de uso?
  • ¿El controlador toma alguna decisión de negocio, o solo traduce y mapea errores?
  • ¿Cada error sale con title, detail y key, y la key es el slug del título?
  • ¿Agregar el siguiente caso de uso es crear una carpeta y enchufar sus rutas, sin abrir las demás?

Si las seis salen limpias, cortaste bien. Si una falla, ya sabes exactamente dónde está la fuga: el checklist te apunta a la capa.

Cortar el código en rebanadas no es una decisión estética. Es decidir, de antemano, dónde podrá doler el cambio y dónde no. El vertical slicing no ordena archivos: ordena decisiones, y mantener una decisión sale siempre más barato que mantener una carpeta. Eso es criterio: no la forma del árbol, sino saber por qué cada cosa quedó donde quedó —y poder explicarlo cuando alguien, en seis meses, te lo pregunte.
